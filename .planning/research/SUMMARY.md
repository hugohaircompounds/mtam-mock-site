# Project Research Summary

**Project:** MTAM Homepage Demo — JTBD-Driven Hair Extension Education Platform Redesign
**Domain:** Single self-contained HTML presentation demo
**Researched:** 2026-03-30
**Confidence:** HIGH

## Executive Summary

This project is a single-file HTML homepage demo built to demonstrate that AI + JTBD (Jobs-to-be-Done) methodology produces more intentional, purposeful design decisions than conventional conversion-rate-optimization approaches. The target is MoreThanAMethod.com — a hair extension education platform — and the demo will be presented in a live business meeting. The output is one clickable `.html` file that can be opened directly in a browser with no build step, no server, and no backend dependencies. The entire value of the demo is not just "what it looks like" but "why each element is where it is," which the JTBD framework supplies.

The recommended approach is a zero-build-step stack: Tailwind CSS v4 Play CDN for styling, GSAP ScrollTrigger for scroll animations, Phosphor Icons v2.1.2 for UI icons, and Google Fonts (DM Serif Display + Manrope) for typography — all loaded from CDN with no compilation. Page architecture follows the Universal Job Map (Define > Locate > Prepare > Confirm > Execute), with each scrollable section mapped to a specific job step. This is not cosmetic organization: the section sequence is derived from how a hair stylist makes a purchase decision, not from how a SaaS homepage template is conventionally ordered. Competitor platforms (NBR Education, Bespoke Hair Education, Harper Ellis) all lead with feature-first or method-centric structures — JTBD-driven structure is a genuine differentiator.

The primary risk is not technical — it is methodological. The single most dangerous failure mode is building a conventionally attractive page and labeling it with JTBD terminology after the fact (Pitfall 3: Job Map as Veneer). Prevention requires deriving the section map from the job map before any HTML is written, then running every headline through the Outcome-Based Messaging lens to eliminate feature language. The secondary risk is presentation-day CDN failure — mitigated by testing the file offline on the meeting device before the event. If both risks are handled correctly, this is a low-complexity, high-impact deliverable.

## Key Findings

### Recommended Stack

The entire stack is CDN-loaded into a single HTML file with no build tooling required. Tailwind v4 Play CDN is the correct choice because it compiles utility classes in-browser at runtime — the "not for production" warning is irrelevant since this file is never deployed. GSAP with ScrollTrigger (now fully free post-Webflow acquisition) produces noticeably more polished animations than AOS and is worth the ~100KB CDN load. Phosphor Icons outperforms Font Awesome Free (no account required) and Lucide (6 weights vs stroke-only) for a beauty/education aesthetic context.

**Core technologies:**
- Tailwind CSS v4 (`@tailwindcss/browser@4`): utility-first styling with zero build step — the only CSS framework that works inline in a single HTML file via CDN
- GSAP v3.14.2 + ScrollTrigger: scroll-driven entrance animations — free, professional-grade, works in all modern browsers without flags
- Phosphor Icons v2.1.2: UI icons in 6 weights (regular + fill sufficient for this demo) — no account, no JS dependency, pure CSS font-face
- Google Fonts — DM Serif Display + Manrope: display serif for hero headings, geometric sans for body; Manrope matches MTAM's existing brand font stack
- CSS custom properties at `:root`: brand token system (colors, fonts, radius, shadow) defined once and referenced throughout all sections

**Offline fallback (if meeting venue has unreliable internet):**
- Replace GSAP with AOS v2.3.1 (8KB vs ~100KB)
- Replace Tailwind CDN with a pre-compiled `<style>` block generated by `npx tailwindcss` against the final HTML
- Base64-encode any critical images

### Architecture Approach

The page is a single vertical scroll organized as a linear job progression. The visitor's scroll position is their position in the decision-making job map — there is no navigation, no tabs, no modal state. Each section is a standalone HTML block separated by comment labels, making the JTBD decision notes artifact easy to produce (one comment per section maps directly to its job step rationale). CSS custom properties are declared once at the top; all sections reference tokens, so a brand color change updates the entire page. JavaScript is limited to GSAP initialization — no state management, no event handlers beyond scroll triggers.

**Major components (in scroll order):**
1. Hero (DEFINE) — outcome headline, subhead identifying audience, single above-fold CTA
2. Proof Bar (LOCATE entry) — social signal strip, numbers, peer presence signal
3. Struggle Acknowledgment (LOCATE framing) — names gatekeeping pain before solution; emotional job address
4. Curriculum Overview (LOCATE content) — outcome-labeled course topics; functional job delivery
5. How It Works (PREPARE) — numbered process steps, reduces setup anxiety
6. Instructor Credibility (CONFIRM — authority) — founder story as struggle-origin narrative, not credential list
7. Testimonials (CONFIRM — peer) — CSS-modeled outcome statements, specific practice changes
8. FAQ (CONFIRM — objection handling) — audience-specific questions, not generic legal boilerplate
9. CTA / Enroll (EXECUTE) — single isolated high-contrast action; no competing elements

### Critical Pitfalls

1. **Vanity design overrides JTBD rationale** — Every section must have a written one-sentence JTBD justification ("This section exists because [job step] requires the visitor to [outcome]") before any HTML is written. If the justification cannot be written, the section does not get built. Prevention happens at architecture phase, not post-build audit.

2. **Feature language masquerading as outcome language** — Run every headline through the Four-Dimensional Job Analysis lens. Warning signs: any headline with a number ("200+ videos"), any headline starting with "Access" or "Get" followed by a product noun, copy that describes the platform before it describes the stylist's transformation. This is a copy-phase failure with LOW recovery cost — but only if caught before the meeting.

3. **Job map as veneer** — The most structurally dangerous pitfall. Section order must be derived from the job map before layout decisions, not retrofitted onto a conventional homepage template. A page that looks like a SaaS template with JTBD labels applied is immediately legible as methodologically dishonest to any audience familiar with the framework. Recovery cost is HIGH (requires structural rebuild).

4. **Single HTML file fails at meeting** — CDN-dependent fonts, icons, and animation libraries that load perfectly on the builder's machine may fail on the presentation laptop with a different browser, resolution, or internet connection. Prevention: test offline on the presentation device before the meeting, verify all assets load with WiFi disabled. Consider inlining critical fonts as base64 if CDN reliability is uncertain.

5. **Emotional and social jobs left implicit** — The functional job (learn extension techniques) is easy to write; emotional job (feel confident, not judged) and social job (be seen as a skilled professional) require dedicated design moments. If these remain as subtext, the anti-gatekeeping brand identity that defines MTAM gets stripped from the demo.

## Implications for Roadmap

Based on research, suggested phase structure:

### Phase 1: Architecture and Section Map
**Rationale:** The most critical pitfall (Job Map as Veneer, Pitfall 3) is a structural failure that cannot be fixed after the page is built without a full rebuild. Section sequence and JTBD justifications must be locked before HTML is written. This is the forcing mechanism that makes the demo credible. ARCHITECTURE.md specifies this must come first; PITFALLS.md assigns it HIGH recovery cost if missed.
**Delivers:** Locked section map with JTBD justification for each section; confirmed page flow (Define > Locate > Prepare > Confirm > Execute); visual hierarchy plan per section; build order sequence.
**Addresses:** All P1 structural features from FEATURES.md; prevents Pitfall 1 (vanity design) and Pitfall 3 (job map as veneer).
**Avoids:** Starting HTML before the section sequence rationale is documented — the single fastest path to a demo that looks done but proves nothing.

### Phase 2: CSS Foundation and Design Tokens
**Rationale:** ARCHITECTURE.md specifies CSS custom properties must be declared first so all subsequent sections reference a single source of truth. Changing a brand token must update the entire page. This phase has no JTBD complexity — it is pure execution — but it gates all subsequent sections.
**Delivers:** Complete `:root` token block (color palette, font families, spacing scale, shadow/radius values); Tailwind v4 `@theme` extension for brand colors; Google Fonts and Phosphor Icons CDN links in `<head>`; verified CDN load order.
**Uses:** Full stack from STACK.md — Tailwind v4, Google Fonts (DM Serif Display + Manrope), Phosphor Icons v2.1.2.
**Implements:** Design system baseline from STACK.md.

### Phase 3: Hero and CTA Sections (Define + Execute)
**Rationale:** ARCHITECTURE.md build order specifies Hero first (hardest copy, most important section) and CTA/Execute second (defines what the visitor is being asked to do — every upstream section points toward it). Writing these two sections first frames all middle sections. The outcome headline is the single highest-leverage copy decision in the entire project.
**Delivers:** Complete Hero section (outcome headline, subhead, above-fold CTA); complete CTA/Enroll section (isolated, high-contrast, single action); confirmed outcome language that passes the Four-Dimensional Job Analysis test.
**Addresses:** Outcome-based hero headline (P1), primary CTA (P1 table stake), EXECUTE job step architecture.
**Avoids:** Pitfall 2 (feature language) — the hero headline is the first and highest-risk copy surface; Pitfall 4 (emotional job left implicit) — hero copy must address functional and emotional jobs simultaneously.

### Phase 4: Locate Sections (Proof Bar, Struggle, Curriculum)
**Rationale:** These three sections form the "evaluate the platform" zone. They must be built together because they share a narrative arc: social signal (Proof Bar) > frustration named (Struggle) > solution revealed (Curriculum). Struggle before Curriculum is the key JTBD sequencing insight — this is where MTAM differentiates from every competitor.
**Delivers:** Proof Bar with social signal numbers; Struggle Acknowledgment / anti-gatekeeping block with problem-first framing; Curriculum Overview with outcome-labeled course topics and icon grid; educator grid with specialization labels.
**Addresses:** Problem-first Struggle Stack section (differentiator), anti-gatekeeping manifesto block (differentiator), course outcomes with outcome language (P1), guest educator grid (P1).
**Avoids:** Pitfall 3 (sections ordered for visual balance rather than job progression); Pitfall 4 (emotional job left implicit — Struggle section is the primary emotional job address).

### Phase 5: Prepare and Confirm Sections (How It Works, Instructor, Testimonials, FAQ)
**Rationale:** These four sections form the trust arc (PREPARE + CONFIRM). ARCHITECTURE.md specifies they must be built in sequence: authority first (Instructor), peer proof second (Testimonials), objection handling third (FAQ). Anti-Pattern 3 from ARCHITECTURE.md explicitly warns against placing testimonials before instructor credibility — peer proof requires an authority foundation to be meaningful.
**Delivers:** How It Works section (3-step process, reduce setup anxiety); Instructor Credibility section (struggle-origin founder story, not credential list); Testimonials with CSS-modeled outcome statements; FAQ with audience-specific questions (texture clientele, prior experience, time commitment).
**Addresses:** Founder identity (P1), outcome-based testimonials (P1), pricing value comparison framing (Confirm zone).
**Avoids:** Pitfall 4 (emotional/social job implicit — testimonials must include transformation language, not just results); Anti-Pattern 3 (testimonials before instructor credibility).

### Phase 6: Pricing Section and JTBD Annotation Layer
**Rationale:** Pricing lands last in the scroll order (Confirm > Execute) because it requires all trust infrastructure to be in place above it. FEATURES.md is explicit: pricing placed before proof of value fails the Confirm job step. The JTBD annotation layer is built after all content sections are finalized — it cannot be added until the sections it annotates are locked.
**Delivers:** Pricing section (3 tiers, effort-reduction framing vs. per-method certification costs, annual savings labeled); JTBD annotation callout labels per section (subtle, not body copy); secondary mid-page CTA (anchor link, no JS).
**Addresses:** Pricing with value comparison (P1), JTBD annotation layer (P1 — demo's unique differentiator), secondary CTA (P2).
**Avoids:** Pitfall 5 (demo tells the story instead of showing it — annotations must be subordinate callouts, not inline explanation text).

### Phase 7: Documentation and Pre-Delivery Testing
**Rationale:** Two of the three required deliverables (JTBD Decision Notes and Prompt List) are documentation artifacts, not HTML. They must be produced alongside the build, not reconstructed after. PITFALLS.md flags both as MEDIUM recovery cost if left until after the build. Pre-delivery testing (offline, presentation resolution) is explicitly required to prevent Pitfall 6.
**Delivers:** JTBD Decision Notes document (section-to-job-step map with one-sentence rationale per section); Prompt List structured by job map phase with JTBD annotations; offline functionality verified on presentation device; outcome language audit pass completed; three-job coverage checklist verified; "Looks Done But Isn't" checklist from PITFALLS.md cleared.
**Addresses:** All items in the PITFALLS.md "Looks Done But Isn't" checklist; Pitfall 6 (file fails at meeting); Pitfall 7 (prompt list reads like a log).
**Avoids:** Delivering a page that passes visual inspection but fails the methodological credibility test when a meeting attendee asks "why is this section here?"

### Phase Ordering Rationale

- Architecture precedes HTML because the structural pitfall (Job Map as Veneer) has HIGH recovery cost and cannot be cosmetically fixed — section sequence must be derived from the job map before any layout decisions.
- CSS foundation precedes all sections because CSS custom properties gate visual consistency across the entire page; changing a token later without this foundation requires manual updates across all sections.
- Hero and CTA are built before middle sections because the outcome headline defines what the visitor is being asked to achieve, and the Execute CTA defines what they are being asked to do — every middle section points toward both ends.
- Locate sections precede Confirm sections because this mirrors the visitor's job progression; building in job-step order also catches narrative gaps between sections during construction rather than during review.
- Pricing and annotation layer are built last because pricing requires all trust infrastructure above it, and annotations can only be finalized after the sections they label are locked.
- Documentation is a parallel and final phase because the Prompt List must be written in real-time during the build; JTBD Decision Notes are best produced section-by-section as each section is finalized; the pre-delivery test is the last gate before delivery.

### Research Flags

Phases with well-documented patterns (skip additional research):
- **Phase 2 (CSS Foundation):** Standard Tailwind v4 + CSS custom properties — official docs are complete, no ambiguity.
- **Phase 3 (Hero + CTA):** Copy framework (Four-Dimensional Job Analysis, Outcome Statement Construction) is fully documented in JTBD reference files.
- **Phase 6 (Pricing layout):** 3-tier pricing card layout is a well-established pattern; value comparison framing is covered in FEATURES.md.

Phases that benefit from careful JTBD reference during build:
- **Phase 1 (Architecture):** Section map derivation requires explicit reference to the Job Map Framework and JTBD for IA — easy to shortcut, critical not to.
- **Phase 4 (Locate sections):** Outcome-language copywriting for course topics requires the Customer Success Language Model; educator grid specialization labels require Top Task Analysis thinking.
- **Phase 5 (Confirm zone):** Testimonial CSS modeling and FAQ question derivation both require explicit use of Customer Success Statement modeling and JTBD interview voice — easy to substitute generic content here.
